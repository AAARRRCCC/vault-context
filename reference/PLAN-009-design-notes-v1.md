# PLAN-009 Design Notes: Discord Utility Bot ("lil helper")

> **This is a working design document, not a finalized plan.** It captures decisions from an early planning session and will likely go through several more revisions before becoming an executable plan. The Worker should not act on this file.

## Name

- Discord display name: **lil helper** (working title, may change)
- Codename in docs: "lil helper" or "the bot"
- Original codename "Homunculus" is retired

## Core Concept Shift

The original PLAN-009 envisioned a bot that acts like a natural conversation participant — reading every message, deciding when to speak via a heuristic gate, and behaving like a member of the friend group. Research and design review led to a fundamental pivot:

**Old model:** Conversational participant with heuristic response gating
**New model:** Utility/service bot with thread-based conversational interface

### Why the pivot

- IBM's Koala study (ACM IUI 2025) found 72% of users preferred a reactive (mention-only) bot over a proactive one. Users called proactive bots "distracting," "intrusive," and "too talkative."
- The heuristic gate was going to be the hardest engineering and the hardest tuning, for a feature that research suggests users don't actually want.
- The target server is low-volume, which means the bot would either respond too rarely to feel present or too often to feel natural.
- Thread-based interaction eliminates the gating problem entirely and gives users explicit control.
- Matt Webb's turn-taking research (Glif/PartyKit) confirms: naive "should I respond?" prompts fail. The problem is fundamentally hard and unsolved for single-bot group chat.

## Architecture Overview

```
Discord Server
│
├── Main Channel (lightweight presence)
│   ├── Scheduled posts (weekly stats, etc.)
│   ├── Command responses (!stats, !features, etc.)
│   ├── @mention responses
│   └── Occasional reactions
│
├── Threads (conversational interface)
│   ├── User opens thread to request feature or ask question
│   ├── Typing debounce: bot waits for silence before responding
│   ├── Multiple users can participate
│   └── Suggestions route to hidden Mayor layer
│
└── Hidden Mayor Layer (user never sees this)
    ├── Sonnet 4.6 as default Mayor brain
    ├── Opus 4.6 reserved for complex/broken situations
    ├── Receives raw user requests from threads
    ├── Does requirements analysis, identifies ambiguities
    ├── Generates 1-3 clarifying questions → relayed to thread
    ├── Finalizes plan + work order → writes to vault-context
    └── Bot confirms scope in plain English to user
```

### User-Facing Flow (what the user sees)

1. User opens thread or posts in existing bot thread
2. User describes what they want ("please make a !stats command that gives message stats")
3. Bot asks 1-3 clarifying questions (generated by hidden Mayor)
4. User answers
5. Bot says something like "Got it, I'll work on that" with a plain-English scope summary
6. Work gets done by Worker/Foreman (unchanged pipeline)
7. Bot reports back: "Done — try !stats"

### Behind the Scenes (what the user doesn't see)

1. User request hits bot in thread
2. Bot sends raw request to Mayor prompt (Sonnet 4.6)
3. Mayor does requirements analysis, identifies ambiguities
4. Mayor generates clarifying questions → bot relays conversationally
5. User answers → fed back to Mayor
6. Mayor writes full plan/WO to vault-context (lil-helper namespace)
7. Worker picks up WO, executes, commits
8. Bot detects completion, reports to user in thread

## Thread Interaction: Typing Debounce

Discord provides `typingStart` gateway events. The bot implements a standard debounce:

- Track active typers per thread as `Set<userId>`
- On `typingStart`: add user to set, reset 5-second timer
- On message send: remove user from set, reset timer
- Discord's typing indicator times out after 10 seconds (natural cleanup)
- Bot only fires API call when: typer set is empty AND debounce timer completes
- Supports multiple users in the same thread naturally

~30 lines of implementation. Prevents the bot from responding mid-thought when someone is composing across multiple messages.

## Segmentation from Existing Infrastructure

lil helper is a **tenant** on infrastructure Brady owns, not a peer of Foreman.

### Repo separation

- Own GitHub repo (not bolted onto Foreman codebase)
- Independent dependencies and deployment
- Separate launchd service, separate process tree
- Bug in lil helper cannot affect Foreman

### vault-context namespacing

- Plans: `lil-helper/plans/`
- Work orders: `lil-helper/work-orders/`
- State: `LIL-HELPER-STATE.md` (separate from main STATE.md)
- Two queues cannot collide

### Trust boundaries

- WO frontmatter tags: `source: lil-helper`, `requested_by: <discord_username>`
- lil helper WOs can ONLY modify files within the lil helper repo
- Any WO that would touch Foreman code, vault-context core files, or anything outside lil helper's directory gets flagged for Brady's approval
- Worker recognizes and enforces these boundaries

### API keys and tokens

- Separate Discord application and bot token
- Separate Anthropic API key (or usage tag for independent spend tracking)
- lil helper budget is isolated from Foreman/Mayor allocation

### Worker scheduling / priority

Worker can't do two things at once. Priority order:

1. Active Brady plans
2. Brady work orders
3. lil helper plans
4. lil helper work orders

Worker checks both state files but always services Brady's queue first.

## Model Selection

- **Thread conversation (user-facing):** Sonnet 4.6 — handles casual back-and-forth, relays Mayor questions
- **Hidden Mayor (planning):** Sonnet 4.6 — requirements analysis, plan writing, WO generation
- **Escalation:** Opus 4.6 — reserved for architecturally complex requests or debugging failures
- **Mayor self-assessment:** Sonnet Mayor should flag when a request might exceed its capability and recommend Opus review before execution

## Suggestion Complexity Tiering

Not every request needs a full Mayor planning session.

- **Simple requests** (add a command, change a response pattern): Fast-track to a single work order. Maybe one clarifying question. No multi-phase plan.
- **Medium requests** (new feature with a few moving parts): Short Mayor session, 1-3 questions, single-phase plan.
- **Complex requests** (new integration, visualization system, architectural change): Full Mayor treatment with multi-phase plan.

The thread conversation should feel like talking to a helpful friend who happens to be a project manager, not like opening a Jira ticket.

## Main Channel Presence

Even as a utility bot, lil helper should have some personality in the shared space:

- Scheduled posts (weekly stats breakdowns, etc.)
- Respond when @mentioned with something that isn't robotic
- Command responses (!stats, !features, !help, etc.)
- Occasional reactions to messages
- Feature registry: `!features` shows all capabilities and who suggested them

Without main-channel presence, users forget the bot exists and don't think to open threads.

## Phase Breakdown (Rough — Needs Further Design)

### Phase 1: Foundation

- Node.js Discord bot (discord.js v14+)
- Thread handler with typing debounce
- Basic command framework (!help, !features)
- @mention response handler
- Channel allowlist
- Logging and cost tracking
- launchd service
- Separate repo scaffolding

### Phase 2: Hidden Mayor Pipeline

- Anthropic API integration (Sonnet 4.6)
- Thread → Mayor → clarifying questions → user → plan/WO flow
- vault-context integration (lil-helper namespace)
- LIL-HELPER-STATE.md management
- WO generation with source tagging and trust boundaries
- Complexity tiering (simple/medium/complex)
- Completion detection and user notification in thread

### Phase 3: Self-Modification + Hot Reload

- Worker picks up lil-helper WOs
- Bot detects changes to its own code/config
- Graceful restart or module reload
- Feature registry (features.json)
- In-thread feedback loop ("Done — try !stats")

### Phase 4+: TBD

- Driven by user suggestions through the pipeline
- Memory/personality may or may not make sense for utility focus
- Stats, visualizations, and other utility features emerge organically
